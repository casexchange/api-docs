## Base URL

- Production: `https://api.casexchange.com/api/v1`
- QA / UAT: Contact support for environment access.
- All endpoints require HTTPS; plain HTTP is rejected.

## Authentication Overview

CaseXchange supports two production-ready authentication methods:

<Columns cols={2}>
  <Card title="JWT Sessions" icon="key" href="/auth/jwt">
    Username/password login issuing JWT access tokens plus HttpOnly refresh cookies. Best for server-to-server or first-party web apps.
  </Card>
  <Card title="OAuth 2.0" icon="user-lock" href="/auth/oauth">
    Authorization Code flow with refresh tokens, OpenID scopes, and discovery metadata.
  </Card>
</Columns>

All API requests must include `Authorization: Bearer <token>`. Tokens encode firm, role, and permissions, which drive authorization checks on every endpoint.

## Versioning

- Current version: `v1`
- Breaking changes trigger a new prefix (`/api/v2`)
- Non-breaking additions (fields, optional params) can ship within v1

## Response Envelope

Every endpoint returns a consistent JSON envelope:

```json
{
  "success": true,
  "data": { /* resource payload */ }
}
```

Errors follow the same structure with an `error` object. See `/concepts/cases#errors`.

## Rate Limiting

- Login endpoints enforce per-email and per-IP throttling.
- Standard API rate limits return headers:
  - `X-RateLimit-Limit`
  - `X-RateLimit-Remaining`
  - `X-RateLimit-Reset`
- Hitting limits returns HTTP `429` with `error.code` set to `rate_limited_*`.
